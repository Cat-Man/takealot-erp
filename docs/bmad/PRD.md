# Product Requirements Document

## 产品名称

Takealot Repricing Platform

## 问题定义

Takealot 卖家需要持续监控某个商品的竞争情况，并在利润受控的前提下快速调整自己的售价。手工查价和手工改价存在三个核心问题：

- 反应慢，无法跟上平台实时价格波动
- 容易误判利润底线，导致低价竞争失控
- 缺少审计记录，无法追溯一次调价为什么发生

## 目标用户

- 中小型 Takealot 卖家
- 管理多 SKU 的运营人员
- 需要控利润、控节奏、控风险的品牌方

## 核心目标

- 获取商品竞争态势，至少能判断是否需要调价
- 根据卖家规则生成建议价
- 在受控条件下执行调价
- 完整记录每次刷新、建议和执行

## 非目标

- 不做通用多平台 ERP
- 不做大而全 BI 系统
- 不把公开网页抓取作为长期核心方案
- 不在未确认 API 能力前假设能拿到全部竞品明细

## MVP 范围

- 本地运营后台
- 商品监控列表
- repricing 规则配置
- 建议价计算
- 调价执行
- 审计记录
- `mock` provider
- `takealot-seller-api` provider 骨架与接入计划

## 成功标准

- 单商品可完成一次“刷新 -> 计算 -> 执行 -> 记录”的闭环
- 规则引擎可稳定保护地板价和最低利润
- 系统可切换 provider，而不改业务层
- 真实接入阶段可基于 Seller API key 完成鉴权

## 当前关键决策

- 主方案从 `browser-first` 调整为 `Seller API-first`
- 浏览器自动化只保留为 fallback
- 竞品最低价能力必须分层建模，不默认等同于“官方 API 必然返回”
